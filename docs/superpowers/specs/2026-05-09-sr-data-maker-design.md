# SR Data Maker Design

## Purpose

SR Data Maker is a Python/PyTorch training-data generation framework for super-resolution and image restoration workflows.
It starts from one source image tree and supports two generation modes:

- Degradation mode: treat the source image as the real target and generate a degraded image.
- Teacher super-resolution mode: treat the source image as the low-quality input and generate a teacher output.
- Restoration mode extension: treat the source image as the input for image-to-image restoration tasks such as denoising, defringing, deblurring, or decompression artifact removal.

The project is designed for long-term extension. New degradation algorithms, learned degradation models, PyTorch super-resolution models, restoration models, and external open-source repositories should be added through registered runners or adapters instead of changing the core pipeline.

## Design Principles

- YAML is the main user-facing control surface.
- The configuration style should feel close to common open-source deep learning projects such as BasicSR and Real-ESRGAN.
- The internal architecture remains modular: source readers, generators, runners, adapters, dataset writers, and orchestration are separate layers.
- The output directory should stay simple and practical for visual comparison.
- The source image tree is not copied into the output dataset by default.
- Generated image directories must preserve the source relative path structure.
- Teacher output directories must include the teacher model name so multiple models can be compared easily.
- Restoration output directories must include the restoration model name for the same reason.
- The manifest is retained as the semantic index and provenance record.
- The code must run on both Windows and Linux.
- GPU selection, precision, tile size, and memory fallback behavior are configured in YAML.
- The first implementation should be a stable single-machine MVP, not a distributed platform.

## Development Workflow Preference

The user preference for this project is direct execution: future commands do not need confirmation before running. Development, inspection, validation, test, formatting, and implementation commands should be run non-interactively when they are needed to complete the current task.

This preference should be reflected in tooling choices:

- Prefer non-interactive CLI commands.
- Do not pause to ask for confirmation before normal project commands.
- Avoid interactive prompts in project scripts unless the user explicitly asks for them.
- Keep command behavior visible through logs, summaries, and generated reports.

## Core Semantics

The source image has no fixed semantic role. Its role depends on the generation task.

In degradation mode:

- `source.role` is `hr`.
- `input` is the generated degraded image.
- `target` references the original source image.
- `target_type` is `real_gt`.
- The source image is not copied into the output dataset in the MVP.

In teacher super-resolution mode:

- `source.role` is `lq`.
- `input` references the original source image.
- `target` is the teacher model output.
- `target_type` is `pseudo_gt`.

The project should avoid calling teacher outputs plain `GT`. They are pseudo targets generated by a model.

In future restoration mode:

- `source.role` is `input`.
- `input` references the original source image.
- `target` is the restoration model output.
- `target_type` is `pseudo_gt` or `restored_output`.
- The output folder is grouped by restoration model name.

## Simplified Dataset Layout

The default dataset layout is intentionally simple:

```text
data/
  sources/
    raw/
      city/day/img001.png
      portrait/a/img002.jpg

  outputs/
    sr_mixed_v1/
      manifests/
        samples.jsonl
        failures.jsonl
        run_summary.json

      degraded/
        degradation_x2/
          city/day/img001.png
          portrait/a/img002.jpg

      teacher/
        RealESRGAN_x2plus/
          city/day/img001.png
          portrait/a/img002.jpg
        SwinIR_x2/
          city/day/img001.png
          portrait/a/img002.jpg

      restored/
        SwinIR_Denoise/
          city/day/img001.png
        DefringeNet_A/
          city/day/img001.png

      logs/
      tmp/
```

Rules:

- `data/sources/raw` is the input image tree.
- `outputs/<dataset_name>/degraded/<task_name>/...` stores degraded images.
- `outputs/<dataset_name>/teacher/<model_name>/...` stores teacher model outputs.
- `outputs/<dataset_name>/restored/<model_name>/...` is reserved for future restoration outputs such as denoising or defringing.
- The path after `<task_name>` or `<model_name>` mirrors the source relative path.
- The source images are not copied into `outputs` by default.
- `manifests` is kept because it records source paths, task names, target type, model names, parameters, seeds, and failures.
- `logs` and `tmp` are runtime support directories and can be disabled or cleaned after generation.

For example, if the source path is:

```text
data/sources/raw/city/day/img001.png
```

Then the generated outputs are:

```text
data/outputs/sr_mixed_v1/degraded/degradation_x2/city/day/img001.png
data/outputs/sr_mixed_v1/teacher/RealESRGAN_x2plus/city/day/img001.png
data/outputs/sr_mixed_v1/teacher/SwinIR_x2/city/day/img001.png
data/outputs/sr_mixed_v1/restored/SwinIR_Denoise/city/day/img001.png
```

This makes comparison straightforward because the same relative path identifies the same source image across source, degradation outputs, teacher outputs, and restoration outputs.

## Manifest Record

Each generated sample is represented as one manifest record. The manifest is not used to create a complicated output structure; it keeps the semantic and reproducibility information that is not visible from folders alone.

Example degradation record:

```json
{
  "sample_id": "degradation_x2::city/day/img001.png",
  "task_type": "superres",
  "generation_mode": "degradation",
  "source": {
    "root": "data/sources/raw",
    "rel_path": "city/day/img001.png",
    "role": "hr"
  },
  "input": {
    "path": "degraded/degradation_x2/city/day/img001.png",
    "role": "lq"
  },
  "target": {
    "source_ref": "city/day/img001.png",
    "role": "target",
    "target_type": "real_gt",
    "storage": "source_reference"
  },
  "outputs": [
    {
      "name": "degraded",
      "path": "degraded/degradation_x2/city/day/img001.png",
      "type": "image"
    }
  ],
  "provenance": {
    "generator": "DegradationGenerator",
    "runner": "ClassicalDegradationRunner",
    "task_name": "degradation_x2",
    "config_hash": "sha256:...",
    "seed": 1234,
    "params": {
      "scale": 2,
      "blur": {"type": "gaussian", "sigma": 1.2},
      "jpeg": {"quality": 70}
    }
  }
}
```

Example teacher record:

```json

    "path": "teacher/RealESRGAN_x2plus/city/day/img001.png",
    "role": "target",

  },
  "outputs": [
    {
      "name": "teacher",
      "path": "teacher/RealESRGAN_x2plus/city/day/img001.png",
      "type": "image",

    "generator": "TeacherSRGenerator",

```

```json
  },

    "config_hash": "sha256:...",
    "seed": 1234
  }

```

Required semantic rules:

- `source.rel_path` must be the path relative to `paths.input_root`.
- Generated files must preserve `source.rel_path` under their output branch.
- `target.target_type` must be either `real_gt` or `pseudo_gt` in the MVP. Future restoration tasks may also use `restored_output`.
- Degradation metadata should be stored in the manifest by default, not as separate per-image JSON files.
- Teacher output folder names default to `model.name`.
- Restoration output folder names default to `model.name`.
- Degradation output folder names default to `task.name`.

## YAML Configuration

The configuration system uses one main YAML file with optional simple `base` inheritance. Later files override earlier base values. The first version should avoid complex interpolation and expression evaluation.

Example:

```yaml
base:
  - ./base/runtime_single_gpu.yaml

name: sr_mixed_v1
model_type: SRDataMaker

runtime:
  device: cuda
  gpu_ids: [0]
  num_workers: 4
  batch_size: 1
  use_fp16: true
  cudnn_benchmark: true
  seed: 1234
  resume: true
  oom_fallback:
    enabled: true
    reduce_tile: true

paths:
  input_root: ./data/sources/raw
  output_root: ./data/outputs/sr_mixed_v1
  weights_root: ./weights
  log_root: ./data/outputs/sr_mixed_v1/logs
  temp_root: ./data/outputs/sr_mixed_v1/tmp

source:
  type: ImageFolderSourceReader
  recursive: true
  exts: [png, jpg, jpeg, webp]

layout:
  preserve_relative_path: true
  copy_source: false
  degraded_root: degraded
  teacher_root: teacher
  restored_root: restored
  filename_policy: keep_source_name

tasks:
  - name: degradation_x2
    enabled: true
    type: DegradationGenerator
    runner:
      type: ClassicalDegradationRunner
    policy:
      mode: repeat_n
      count: 2
    degradation:
      scale: 2
      blur:
        enabled: true
        type: gaussian
        kernel_size: 9
        sigma: [0.2, 2.0]
      resize:
        enabled: true
        mode: [bilinear, bicubic]
      noise:
        enabled: true
        type: gaussian
        std: [1, 10]
      jpeg:
        enabled: true
        quality: [30, 95]
    output:
      root: degraded
      folder_name: degradation_x2
      target_type: real_gt
      save_params_to_manifest: true

  - name: teacher_sr_realesrgan
    enabled: true
    type: TeacherSRGenerator
    runner:
      type: PyTorchTeacherRunner
    policy:
      mode: once
    model:
      type: RealESRGAN
      name: RealESRGAN_x2plus
      weights: ./weights/realesrgan_x2plus.pth
      scale: 2
      tile: 512
      tile_pad: 16
      pre_pad: 0
      half: true
    output:
      root: teacher
      folder_name: RealESRGAN_x2plus
      target_type: pseudo_gt

  # Future extension example. This is not required in the MVP.
  - name: denoise_swinir
    enabled: false
    type: RestorationGenerator
    runner:
      type: SwinIRDenoiseRunner
    policy:
      mode: once
    model:
      type: SwinIRDenoise
      name: SwinIR_Denoise
      weights: ./weights/swinir_denoise.pth
      tile: 512
      half: true
    output:
      root: restored
      folder_name: SwinIR_Denoise
      target_type: restored_output

manifest:
  save_jsonl: true
  save_summary: true
```

Configuration sections:

- `runtime` controls device, GPU IDs, precision, workers, seeds, resume, and memory fallback.
- `paths` controls input, output, weights, logs, and temporary directories.
- `source` configures input data reading.
- `layout` controls the simplified mirrored output layout.
- `tasks` defines generation tasks. Each task declares its generator type, runner type, task parameters, policy, and output location.
- `degradation` contains degradation parameters.
- `model` contains teacher super-resolution or restoration model parameters.
- `manifest` controls manifest writing.

Task and parameter switches:

- Every task supports `enabled: true | false`.
- A disabled task is skipped and does not need its runner or weights to be available.
- Degradation operators support their own `enabled` switches, so blur, resize, noise, and JPEG can be turned on or off independently.
- Real-ESRGAN teacher parameters such as `weights`, `scale`, `tile`, `tile_pad`, `pre_pad`, and `half` are read from YAML.
- The MVP teacher task uses Real-ESRGAN 2x by default: `model.name: RealESRGAN_x2plus` and `scale: 2`.

## GPU And Platform Requirements

The project must run on both Windows and Linux.

Implementation requirements:

- Use `pathlib.Path` for all filesystem paths.
- Do not put Linux-only shell scripts on the core execution path.
- Wrap external repository calls through Python `subprocess` APIs.
- Keep temporary directories, logs, caches, and outputs configurable through YAML.
- Do not assume symlink support because the simplified MVP does not require links.
- Do not hard-code CUDA device IDs in code.

GPU behavior:

- `runtime.gpu_ids` chooses visible CUDA devices for the current run.
- The first MVP maps `[0]` to `cuda:0` inside PyTorch.
- V100 and RTX 5070 Ti should be handled through configuration differences, not code branches.
- Tile size, batch size, half precision, and OOM fallback behavior are configurable.
- If CUDA is unavailable and `runtime.device` is `cuda`, validation should fail before generation starts.

## Architecture

The Python package should use a `src` layout:

```text
src/
  sr_data_maker/
    config/
      loader.py
      merge.py
      validator.py
    core/
      types.py
      protocols.py
      registry.py
      exceptions.py
      context.py
    sources/
      image_folder.py
    generators/
      degradation.py
      teacher_sr.py
      restoration.py
    runners/
      degradation/
        classical.py
      teacher/
        pytorch_runner.py
      restoration/
        pytorch_runner.py
    adapters/
      degradation/
      teacher/
      restoration/
    dataset/
      writer.py
      manifest.py
      naming.py
    orchestration/
      executor.py
      orchestrator.py
      state_store.py
      retry.py
    validation/
      config_validator.py
      output_validator.py
      manifest_validator.py
    cli/
      main.py
      commands/
        run.py
        validate.py
        inspect.py
```

Core responsibilities:

- `SourceReader` reads source records and computes `rel_path` from `paths.input_root`.
- `Generator` defines sample-generation semantics.
- `Runner` executes one algorithm or model.
- `Adapter` wraps external repositories or non-standard model APIs as runners.
- `RestorationGenerator` is reserved for non-super-resolution image-to-image restoration tasks such as denoising or defringing.
- `DatasetWriter` writes generated images and manifests using the mirrored relative path layout.
- `PipelineExecutor` drives the run.
- `RunStateStore` tracks completed, skipped, and failed source-task pairs for resume.

## Interfaces

The MVP should keep interfaces small.

```python
class SourceRecord:
    source_id: str
    path: str
    rel_path: str
    meta: dict
```

```python
class SourceReader:
    def iter_sources(self):
        ...
```

```python
class RunnerOutput:
    outputs: dict
    meta: dict
```

```python
class Runner:
    name: str

    def run(self, inputs: dict, context):
        ...
```

```python
class Generator:
    name: str

    def generate(self, source: SourceRecord, context):
        ...
```

`Generator` owns the manifest semantics. `Runner` owns execution. `DatasetWriter` owns output path generation, file writing, and manifest persistence.

## Execution Flow

The runtime flow is:

1. Load and merge YAML configuration.
2. Validate configuration, paths, registry entries, CUDA availability, and output semantics.
3. Build source reader, generators, runners, dataset writer, and run state store.
4. Iterate source records and compute stable `rel_path` values.
5. For each source and task, check resume state.
6. Run the generator and its runner.
7. Build output paths by joining output branch, task or model folder, and `source.rel_path`.
8. Validate generated output files and manifest record.
9. Write generated images and append `samples.jsonl`.
10. Record failures in `failures.jsonl`.
11. Write `run_summary.json`.

Failures outside configuration validation should be isolated to the sample or source-task level when possible.

Supported run modes for MVP:

- `resume`: skip completed source-task pairs.
- `force_rerun`: ignore previous state and regenerate.
- `retry_failed`: rerun only failed records.

The first implementation may keep state in JSONL files, but the design should allow replacing it with SQLite later.

## MVP Scope

The first implementation includes:

- `ImageFolderSourceReader`
- `DegradationGenerator`
- `TeacherSRGenerator`
- `RestorationGenerator` interface reservation for future image-to-image restoration tasks
- `ClassicalDegradationRunner`
- `PyTorchTeacherRunner`
- `DatasetWriter`
- `PipelineExecutor`
- YAML config loader with simple `base` inheritance
- Registry-based component construction
- CLI commands: `run`, `validate`, `inspect`
- `samples.jsonl`, `failures.jsonl`, and `run_summary.json`
- Mirrored output directories for degraded images and teacher images
- A reserved `restored/<model_name>/<source_rel_path>` layout for future restoration models
- Resume support
- Windows and Linux path compatibility

The first implementation excludes:

- Copying source images into the output dataset
- The previous `artifacts` and `views` directory layers
- Distributed execution
- Multi-node scheduling
- Complex multi-GPU scheduling
- Web UI
- Large adapter library
- Learned degradation networks
- Full restoration model implementations beyond the reserved interface and config shape
- Automatic model downloading
- Complex reporting dashboards
- Full SQLite state store

## Validation Strategy

Startup validation should catch:

- Invalid YAML structure
- Missing paths
- Missing weights for model tasks
- Unknown `type` values
- Unknown runners
- Invalid `target_type`
- Mismatched task type and output semantics
- CUDA unavailable when requested
- Output directory permissions
- Invalid output folder names

Runtime output validation should catch:

- Missing generated files
- Unreadable generated images
- Invalid image dimensions
- Missing manifest fields
- Invalid output paths
- Duplicate sample IDs
- Output paths that do not preserve the source relative path

The `inspect` command should report:

- Number of source images
- Number of generated samples
- Count by generation mode
- Count by target type
- Count by degradation task
- Count by teacher model
- Count by restoration model when restoration tasks are present
- Failure count
- Size distribution of source and generated images
- Missing output count

## Testing Strategy

The first test suite should focus on the framework before real models:

- Unit tests for config loading and base merge behavior.
- Unit tests for registry construction.
- Unit tests for manifest record validation.
- Unit tests for mirrored output path generation.
- Integration test with a nested image folder, mock degradation runner, and mock teacher runner.
- Interface-level test for a mock restoration runner can be added when `RestorationGenerator` is implemented.
- Resume test that rerunning the same config skips completed work.
- Cross-platform path tests using `pathlib` behavior.

Real open-source model adapters should have separate smoke tests because they require weights and GPU-specific dependencies.

## Implementation Order

Recommended implementation order:

1. Project skeleton and packaging.
2. Config loader, merge logic, and validator.
3. Core types, protocols, registry, and context.
4. Image folder source reader with nested directory traversal and `rel_path` generation.
5. Dataset writer, naming, manifest writer, and mirrored output path writer.
6. Mock or minimal classical degradation runner.
7. Degradation generator.
8. Minimal PyTorch teacher runner and teacher generator.
9. Pipeline executor and run state.
10. CLI commands.
11. Inspect command and basic summary reports.
12. Tests and example configs.
13. Optional follow-up: `RestorationGenerator` and a mock restoration runner.

## MVP Decisions

- Source images are not copied into `output_root` by default.
- Degraded outputs are written to `degraded/<task_name>/<source_rel_path>`.
- Teacher outputs are written to `teacher/<model_name>/<source_rel_path>`.
- Future restoration outputs are written to `restored/<model_name>/<source_rel_path>`.
- Teacher folder names default to `model.name`; task config may override them with `output.folder_name`.
- Restoration folder names default to `model.name`; task config may override them with `output.folder_name`.
- Degradation folder names default to `task.name`; task config may override them with `output.folder_name`.
- The first real super-resolution teacher adapter should target Real-ESRGAN 2x, using `RealESRGAN_x2plus` as the default model name and `scale: 2`.
- Degradation uses conventional operators in the MVP: resize, blur, noise, and JPEG compression.
- Degradation and teacher tasks can be independently enabled or disabled from YAML.
- The initial state store should use JSONL files. SQLite remains a planned replacement once dataset scale makes lookup speed a real bottleneck.
